# Context Mesh Framework Reference

## Overview
Context Mesh is a documentation and knowledge management framework for software projects. It enforces strict separation of concerns between business intent, technical decisions, and implementation patterns. The framework is designed to support AI agents and human teams in planning, building, and learning from software projects.

### 3 Steps: Intent → Build → Learn
1. **Intent**: Define what the project/feature does and why (business/user value).
2. **Build**: Make technical decisions and implement using documented patterns.
3. **Learn**: Capture outcomes, insights, and update context as the project evolves.

## File Type Separation Rules
- **Feature files** (`feature-*.md`): WHAT and WHY only. No technical details. No code. No library names. Must link to relevant decisions and patterns.
- **Decision files** (`decisions/*.md`): Technical choices, rationale, alternatives, outcomes. No business value. No code examples.
- **Pattern files** (`knowledge/patterns/*.md`): Code patterns, implementation examples, how-to guides. No business value. No decision rationale.
- **Agent files** (`agents/`): Reserved for agent-specific context and instructions.
- **Anti-patterns** (`knowledge/anti-patterns/`): Document what NOT to do.

## Plan, Approve, Execute Pattern (MANDATORY)
- **Plan**: Define intent and required decisions before implementation.
- **Approve**: Ensure decisions and patterns are documented and approved.
- **Execute**: Implement according to approved context and update documentation.

## When to Create Files
- Create a **feature file** for each business capability or user-facing module.
- Create a **decision file** for each major technical choice, library, or architecture.
- Create a **pattern file** for each reusable code structure or implementation approach.
- Do NOT duplicate information across files. Link instead.

## Definition of Done
- All features, decisions, and patterns are documented and linked.
- Bidirectional links between features, decisions, and patterns.
- Status section in every file (created, status, phase).
- Changelog updated after any change.

## Bidirectional Links Rules
- Features must link to relevant decisions and patterns.
- Decisions must link to related features and patterns.
- Patterns must link to related decisions and features.
- Use relative markdown links.

## Status Section Requirements
- Every file must have a status section with creation date, phase, and status.
